# Story 1.3: Build Pipeline

Status: done

## Story

As a contributor,
I want a build pipeline that assembles compiled agent skill files from authoring sources,
So that I can verify my agent + skill combination produces correct output and CI catches any direct edits to `compiled/`.

## Acceptance Criteria

1. **Given** `agents/planner-agent.md` contains `### build-study-plan` under `## Skills` and `skills/build-study-plan.md` exists **When** I run `bash scripts/build-skills.sh` **Then** `compiled/bmad-study-planner.md` is created with the full content of `skills/build-study-plan.md` inlined under `## Skills`

2. **Given** `compiled/bmad-study-planner.md` was generated by `build-skills.sh` **When** a contributor edits `compiled/bmad-study-planner.md` directly and opens a PR **Then** CI runs `build-skills.sh`, diffs the output against the committed file, finds a mismatch, and fails with a message: `"compiled/ must not be edited directly — run build-skills.sh"`

3. **Given** `package.json` has `"prepublishOnly": "bash scripts/build-skills.sh"` **When** `npm publish` is run **Then** `build-skills.sh` executes before the package is published, ensuring `compiled/` is always up to date at publish time

4. **Given** `tests/agents/structure.test.ts` exists and `vitest run` executes **When** it runs against any file in `compiled/*.md` **Then** for each compiled agent file, the test asserts: (a) YAML frontmatter is present with `id`, `version`, and `type` fields, (b) `## On Activation` section exists, (c) `## Session Workflow` section exists, (d) `## After Session` section exists, (e) the string `memory-protocol-version` appears in the file body

5. **Given** `build-skills.sh` runs on a CI agent with a clean checkout **When** it completes **Then** it exits `0` and `compiled/` contains exactly one `.md` file per agent file in `agents/`

## Dev Notes

### Build Script Algorithm (Pattern 1 — Skill Composition)

Per `architecture.md#Pattern 1`, skills are inlined under `## Skills` with their YAML frontmatter stripped. The `### skill-name` heading is kept; content from `skills/{skill-name}.md` (minus frontmatter) is placed under it, replacing any placeholder content in the source agent file.

**State machine for `build_agent` function:**
- `frontmatter`: copy lines until second `---` → switch to `normal`
- `normal`: copy lines; on `## Skills` → switch to `skills`
- `skills`: copy `## Skills` and `### skill-name` headings; on `### skill-name`, inline skill file (strip frontmatter), switch to `skill_placeholder`; on `## non-Skills` heading → switch to `normal` and copy
- `skill_placeholder`: skip placeholder lines; on `### next-skill` → inline next skill and stay in `skill_placeholder`; on `## heading` → switch to `normal` or `skills` and copy

**Frontmatter stripping for skills:**
```bash
awk 'BEGIN{skip=1;count=0} /^---/{count++;if(count==2){skip=0;next};next} !skip{print}' "$skill_file"
```

### `--check` Mode (AC #2)

Builds to a temp directory, then diffs against the current `compiled/` directory. If `compiled/` doesn't exist or is empty (clean CI checkout), the check is skipped (no committed files to compare). Only fails when `compiled/*.md` files exist AND differ from what the build produces.

### compiled/ is Gitignored

Per Story 1.1, `compiled/` is in `.gitignore`. The `--check` mode catches local workspace mismatch (e.g., if someone manually edited compiled/ without committing). In CI, the build step regenerates compiled/ from source, ensuring tests always run against freshly-built output.

### Test Strategy

The `beforeAll` in `structure.test.ts` runs `build-skills.sh` to ensure compiled/ is always fresh when tests execute. This means:
- Tests are always authoritative (no stale compiled/ state)
- The test suite implicitly validates that `build-skills.sh` exits 0 (it would throw if the script fails)

### tsconfig Update

Story 1.1 deferred the tsconfig `include` covering `tests/` (see Story 1.1 Dev Notes deferred finding). This story resolves that: add `"tests/**/*"`, plus `moduleResolution: node`, `esModuleInterop: true`, `skipLibCheck: true`, and `types: [node, vitest/globals]`. `skipLibCheck` is required because Vite's type declarations (a transitive dep of Vitest) reference `rollup/parseAst` which requires `node16`+ module resolution.
